# Content Schema

Defines all content types in this project, their fields, relationships, and i18n strategy.
Generated by `astro-builder`. Update by re-running `/astro-builder:init` or editing directly.

## Content types

### Docs

**Collection name**: `docs`
**Folder**: `src/content/docs/`
**Format**: MDX
**Translatable**: yes (locale subfolders; English only in v1)
**URL pattern**: `/docs/{slug}` (default locale unprefixed; a future pt-BR becomes `/pt-BR/docs/{slug}`)

#### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `title` | `string` | yes | Display title |
| `description` | `string` | yes | Short summary (used in meta, cards, and the docs index) |
| `sidebarLabel` | `string` | no | Short label for the sidebar when the title is long |
| `order` | `number` | yes | Sort order within a sidebar group |
| `group` | `enum` | yes | `"Getting started"` \| `"Core concepts"` \| `"Reference"` |
| `tags` | `string[]` | no | Flexible tags (no fixed enum); defaults to `[]` |
| `lang` | `enum` | yes | Locale code (`en`) |
| `translationKey` | `string` | yes | Links the same doc across locales |
| `draft` | `boolean` | no | `true` = excluded from build, listings, and sitemap; defaults to `false` |

#### Relationships

None in v1. Docs are standalone reference pages, grouped and ordered by `group` + `order`.

## Cross-collection relationships

None in v1. A dated `blog` collection (with `date` + RSS) is **deferred to phase 2**. Because no
dated collection exists, the RSS endpoint and `<link rel="alternate" type="application/rss+xml">`
are intentionally omitted — they return with the blog.

## Frontmatter example

```mdx
---
title: "Introduction"
description: "What Scout is, the problem it solves, and how a plain-language scenario becomes a verified, self-healing check."
sidebarLabel: "Introduction"
order: 1
group: "Getting started"
tags: ["overview", "concepts"]
translationKey: "introduction"
lang: en
draft: false
---

Content body starts here…
```

## i18n strategy

- Each translatable content type has locale subfolders: `docs/en/` (and `docs/pt-BR/` later).
- Files use the same slug across locales, identified by `translationKey`.
- The `lang` field is the canonical locale identifier — never parse the path.
- `draft: true` entries are excluded from `getStaticPaths()` in production.
- The default locale (`en`) is **unprefixed** in URLs (`prefixDefaultLocale: false`); the slug
  passed to routes strips the `en/` loader-id prefix.
