# Style Guide

Defines the voice, tone, and writing conventions for Scout.
Generated by `astro-builder`. Update by re-running `/astro-builder:init` or editing directly.

> This file is the source of truth for the site's voice, copy, and the honesty guardrails below.

## Voice and tone

**Voice** (consistent, brand identity): editorial-scientific, calm, precise — an instrument's
voice. It reports a *reading*; it does not hype. Three physical-object words: **calibrated,
optical, measured**. Mission first, feature second.

**Tone** (varies by context):
- **Docs / main content**: explanatory and unhurried. Lead with the outcome, then the mechanism.
- **UI copy** (buttons, labels, nav): plain verbs, sentence case, no exclamation. "Read the docs",
  not "Get started now!".
- **Error messages**: clear, helpful, never blame the user.
- **CTAs**: action-oriented, specific (not "Click here").

## Audience

Developers and AI coding agents adopting Scout for browser QA. They evaluate tools fast and
distrust marketing. Write for someone who will run the command, not just read about it.

Assume the reader: is technical, knows what CI and a browser test are, and wants the mechanism and
the honest tradeoff — not a pitch.

## The thesis (sell this, in this order)

The bottleneck moved: code is cheap now that agents write it; judgment and taste stay scarce.
Scout is the quality gate for the age of agents — delegate the build, keep the outcome in focus.

### Honesty guardrails (do NOT break)

- **Determinism / "0 LLM on replay" is NOT the differentiator.** Mention cheap replay as a
  supporting nicety, never the headline moat.
- **The real, defensible moat is the combination:** no code artifact to own · zero-setup
  standalone binary · unattended runtime self-heal · the gate for autonomous loops
  (`scout report --check`) · profiles + i18n first-class.
- Playwright Agents moving the same direction is **tailwind, not threat** — pitch the moat.
- Cost framing when needed: "pay once on the first run (~$0.10–0.30 frontier), replay free."

## Writing rules

### Sentences
- Plain verbs. Short. One idea per sentence.
- Active voice by default. Passive only for deliberate emphasis.

### Paragraphs
- 2–4 sentences. Lead with the most important information (inverted pyramid).
- No filler openers ("In today's world…", "It's important to note that…").

### Headings
- Sentence case: "How a scenario runs" (not "How A Scenario Runs").
- H2 for major sections, H3 for sub-topics. Rarely deeper.
- Scannable: the page structure should be clear from headings alone.

### Lists
- Bullets for unordered items; numbers when sequence matters (e.g. the verify → replay → heal cycle).
- Parallel structure across items.

### Technical terms
- Define jargon on first use. Keep `scout` command names in `code`.
- Never translate code; do not invent acronyms.

### Numbers and dates
- Numbers 1–9 spelled out; 10+ numerals. Stat readouts (`0`, `1 sentence`) stay numeric on purpose.
- Dates: locale-aware via `formatDate(date, locale)` from `@lib/format`. Never hardcode date formats.

## Content structure

### Docs pages

1. **Outcome** — what this page lets the reader do, in one sentence.
2. **Context** — why it matters / where it fits.
3. **Body** — the mechanism, step by step.
4. **Takeaway** — the command to run or the next page.

No mandatory word count. The right length is the minimum needed to be useful.

### Metadata (frontmatter)

- `title`: under 60 characters, descriptive.
- `description`: 120–160 characters, a complete sentence (used in meta + cards).
- `sidebarLabel`: short label for the sidebar when the title is long.
- `order`: integer for sidebar sorting within a `group`.
- `group`: one of "Getting started" | "Core concepts" | "Reference".
- `tags`: 2–5, reuse existing tags first.

## What to avoid

- Hype, exclamation marks, and superlatives ("blazing fast", "magical").
- Selling determinism / "0 LLM on replay" as the moat (see honesty guardrails).
- Filler phrases and excessive hedging ("might", "could potentially").
- Passive voice as a default; jargon without a first-use definition.
